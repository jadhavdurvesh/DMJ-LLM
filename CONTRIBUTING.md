# 🤝 Contributing to DMJ LLM

First off, thank you for your interest in contributing to **DMJ LLM**! 🎉

Whether you're fixing a typo, improving documentation, reporting a bug, or contributing code, your help is appreciated. Every contribution helps make the project better.

---

# 🌟 Ways to Contribute

You can contribute in many ways, including:

- 📚 Improving documentation
- 🐛 Reporting bugs
- 💡 Suggesting new features
- 🔧 Fixing bugs
- ⚡ Improving performance
- 🧪 Writing tests
- 🧠 Enhancing datasets
- 💻 Contributing code
- 🎨 Improving UI/UX
- 🌍 Translating documentation

---

# 🚀 Getting Started

## 1. Fork the Repository

Click the **Fork** button on GitHub to create your own copy of the repository.

## 2. Clone Your Fork

```bash
git clone https://github.com/<your-username>/DMJ-LLM.git
cd DMJ-LLM
```

## 3. Create a Branch

Never work directly on the `main` branch.

```bash
git checkout -b feature/your-feature-name
```

Examples:

```text
feature/tokenizer
feature/training-pipeline
docs/model-architecture
fix/dataset-validation
```

---

# 💻 Development Guidelines

Please follow these guidelines when contributing:

- Keep pull requests focused on a single topic.
- Write clear and descriptive commit messages.
- Keep code clean and readable.
- Add documentation for new features.
- Update existing documentation when behavior changes.
- Test your changes before submitting.

---

# 📝 Commit Message Convention

Use descriptive commit messages.

Examples:

```text
feat: add tokenizer module
fix: resolve dataset validation bug
docs: improve training guide
refactor: simplify preprocessing pipeline
chore: update dependencies
```

---

# 🔀 Pull Request Process

1. Update your branch with the latest changes.
2. Push your branch to your fork.

```bash
git push origin feature/your-feature-name
```

3. Open a Pull Request.

Include:

- Summary of the changes
- Purpose of the contribution
- Screenshots (if applicable)
- Related issue (if applicable)

---

# 📚 Documentation

If your contribution changes functionality, please update the relevant documentation.

Documentation is located in the `docs/` directory.

---

# 🐛 Reporting Bugs

When reporting a bug, please include:

- Operating System
- Python version
- Steps to reproduce
- Expected behavior
- Actual behavior
- Screenshots or logs (if available)

---

# 💡 Feature Requests

Feature requests should include:

- A clear description
- Motivation
- Possible implementation ideas
- Expected benefits

---

# 📏 Coding Standards

Please follow these general principles:

- Write readable code.
- Use meaningful variable names.
- Keep functions focused on one responsibility.
- Avoid unnecessary complexity.
- Document public APIs and important functions.
